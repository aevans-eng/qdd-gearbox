### [session] — Cycloidal Gear Parameter Strategy (Decision Pending)

**What:** Transitioning skeleton from involute gear parameters (R30/P11/S8, module=2.5mm) to cycloidal gear parameters (R48/P18/S12, pygeartrain). Key insight: gear_module and classic pitch diameter formulas don't apply to cycloidal teeth. For cycloidal gears, use outer diameter (tip circle) from script output instead of `m×N`, and tooth depth is computed from profile vertex geometry, not `2.25×m`.

**Why:** Preparing to import `gearbox_CAD.step` (run 001) into the skeleton assembly. Skeleton params need to reflect actual cycloidal geometry before STEP import so bearing pockets land correctly.

**Decision made (after param dump audit):** gear_module appears in exactly 2 formulas. Remove it. Also found that ring_gear_OD is currently wrong (101.25mm skeleton vs 95mm STEP) because Formula.71 double-counts tooth dedendum — the script's ring_wall_thickness already includes it. Formula.17 also needs deletion: it derives planet_mid_path from (sun_PD + planet_PD)/2, which is only valid for involute gears where pitch circles are tangent. For cycloidal, carrier radius is a pygeartrain output, not derivable from outer diameters.

**4 changes required in CATIA skeleton:**
1. Formula.71: `ring_pitch_diam + 2*(1.25*gear_module + ring_wall_thickness)` → `ring_pitch_diam + 2*ring_wall_thickness` (ring_gear_OD: 101.25 → 95mm)
2. Formula.17: delete — set `planet_mid_path = 23.0mm` directly (was incorrectly formula-derived at 23.725mm)
3. Carrier cutout formula (external part): `planet_pitch_diam + 2*gear_module + carrier_planet_clearance` → `planet_outer_diam + carrier_planet_clearance`
4. Delete `gear_module` parameter

**Cascade from ring_gear_OD fix:** housing_ring_gear_bore 101.55 → 95.3mm ✓, housing_width 115.25 → 109mm, lid_step_OD 101 → 94.75mm.

### [20:43] — Gears Product Architecture for STEP Integration

**What:** Built a new product-level structure in CATIA (`gears`) to handle the STEP-to-parametric workflow. Four sibling parts inside the product:

| Part | Role |
|------|------|
| `skel_ref` | Fixed reference part — holds pasted skeleton planes, sketches, axes. Everything mates to this. |
| `gear_set` | Original placeholder gears (simple pads). Still drives downstream parametric parts (carrier, etc.). |
| `gear_part_output_rev01` | Imported STEP from pygeartrain (real herringbone cycloidal tooth geometry). |
| `cutting/adding bodies` | Separate part for boolean operations (bearing bores, bolt holes, etc.). |

**Key decision — external params per part, not centralized in skel_ref:** Each part that needs skeleton data independently pulls publications from the master skeleton via Paste Link. Cross-referencing external params between parts within the product was unreliable — broken links when editing. Direct skeleton → part references are more stable.

**Why this structure:**
- Cutting bodies are **not dependent on any specific gear part** — they reference skeleton datums only. This means they survive STEP swaps and can be reused across revisions.
- Modular iteration — can update the STEP import, adjust cutting bodies, or tweak gear_set independently without cascading breakage.
- skel_ref acts as the single fixed anchor — all parts constrain to it, not to each other.
